The Problem

A content team writing for 27 clients — manually

A digital marketing agency managed SEO content for 27 client websites across multiple industries: solariums, electronics repair, construction, locksmith services, automotive, cleaning. Each client required articles in their own language (French or English), their own tone, about their own products.

Writing was done by hand. Research, draft, manual SEO check, WordPress formatting, Elementor layout, Yoast meta — repeat for every article, every client, every month. At 2 articles per month across 27 clients, this was hundreds of hours of rule-based, repeatable work.

"The bottleneck wasn't creativity. It was the time it took to run the same SEO checklist, in two languages, for dozens of clients, every single month."

What was slowing them down

  • Manual topic research per client, per article
  • SEO checked by eye — inconsistent quality
  • WordPress layout built manually every time
  • No system to prevent duplicate topics
  • Bilingual content meant double the work

What they needed

  • One tool handling all 27 clients
  • Automated SEO validation before publishing
  • Direct WP publishing with full layout
  • Human editorial control kept — no auto-publish
  • Zero learning curve for the team
The Solution

Albert — an agent that lives in Slack

⚠ Client name anonymized under NDA — architecture and code are original work

I built Albert: a Python agent the content team talks to entirely through Slack. No interface to learn, no dashboard to open, no file to upload. The editor types @Albert, makes five decisions, and eight minutes later a fully formatted WordPress draft is ready for review.

Albert handles everything in between: scraping the client's site for real internal links, querying Google for actual search intent, checking competitor blogs for content gaps, writing a 1,200–1,950 word article in the client's exact voice, running 19 SEO checks, correcting what fails, building the Elementor layout, injecting Yoast meta, and posting the draft to WordPress.

Engineering

What I built

The system is a modular Python pipeline — 12 files, each with a single responsibility. The Slack layer has zero business logic. The pipeline has zero Slack imports. This separation means the same agent can be triggered from a scheduler, a CLI, or any future interface without touching the core logic.

The SEO validator

19 rules measured in pure Python — word count, keyword density, H1/H2 presence, meta description length, internal links, image placeholders, FAQ count, em-dashes, and more. The model never self-validates. If the article fails, a surgical correction engine applies fixes in order of blast radius (metadata first, then insertions, then prose rewrites) and re-validates. Max 2 rounds. Remaining failures are surfaced to the editor — never silently swallowed.

The research engine

Before suggesting topics, Albert runs 4 parallel scrapers via Playwright: the client's WordPress blog (to prevent duplicate topics), the client's live navigation (for real internal URLs — never invented), Google's "People Also Ask" (for real search intent), and competitor blogs (for content gap analysis). All 4 run concurrently — total research time is bounded by the slowest single call.

The WordPress publisher

Every draft lands in WordPress with a complete Elementor layout — not raw HTML. The editor sees the article exactly as it will look live. Yoast meta description is pre-filled. JSON-LD schemas (BlogPosting, FAQPage, HowTo) are injected. A CTA button links to the client's contact page.

Per-client configuration — zero hallucination

Each of the 27 clients has a JSON profile: name, industry, language, region, tone, vocabulary to use and avoid, products, seasonal calendar, WordPress credentials. Albert can only write facts that exist in this file. No product, URL, or claim is ever invented.

Reflection

What this project taught me

Separate the model from the logic

The biggest architectural decision was making the LLM call provider-agnostic from day one. The model is called in exactly one function. Everything else — validation, correction, publishing — is deterministic Python. This made the system reliable and testable independent of model behavior.

Never trust the model to validate itself

All 19 SEO rules are measured in code, not asked of the model. This made every failure deterministic, reproducible, and fixable without touching the prompt.

The interface is a product decision

Choosing Slack as the interface meant zero onboarding for the content team. They didn't need to learn a new tool. Keeping humans in their existing environment was as important as the technical architecture.

Config is the hallucination prevention system

The JSON profile per client isn't just configuration — it's the safety layer. By defining a strict boundary of what Albert can say about each client, factual accuracy became a structural guarantee, not a prompt engineering hope.
