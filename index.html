The Problem

A content team writing for 27 clients by hand, every single month

A digital marketing agency managed SEO content for 27 client websites across wildly different industries: solariums, electronics repair, construction, locksmiths, automotive, cleaning. Each client wanted articles in their own language (French or English), their own tone, about their own products.

Writing was done by hand. Research, draft, manual SEO check, WordPress formatting, Elementor layout, Yoast meta — repeat for every article, every client, every month. At two articles per month across 27 clients, this was hundreds of hours of rule-based, repeatable work the team was doing instead of strategy.

The bottleneck wasn't creativity. It was the time it took to run the same SEO checklist, in two languages, for dozens of clients, every single month.

What was slowing them down

  • Manual topic research per client, per article
  • SEO checked by eye — inconsistent quality across writers
  • WordPress layout built manually every single time
  • No system to prevent duplicate topics across months
  • Bilingual content meant double the workload

What they needed

  • One tool handling all 27 clients without per-client code
  • Automated SEO validation before anything reaches WordPress
  • Direct WP publishing with full Elementor layout — not raw HTML
  • Human editorial control kept — no auto-publish, ever
  • Zero learning curve for the content team
The Solution

Albert — an agent that lives in Slack

NDA notice: client name anonymized. Architecture, code, and pipeline are original work.

I built Albert: a Python agent the content team talks to entirely through Slack. No interface to learn. No dashboard to open. No file to upload. The editor types @Albert, makes five quick decisions, and roughly eight minutes later a fully formatted WordPress draft is waiting for review.

Albert handles everything in between: scraping the client's site for real internal links, querying Google for actual search intent, checking competitor blogs for content gaps, writing a 1,200–1,950 word article in the client's exact voice, running 19 SEO checks, correcting what fails, building the Elementor layout, injecting Yoast meta, and posting the draft to WordPress.

  1. @Albert in any Slack channel

    Albert responds with a client dropdown. Nothing to install. No login required.

  2. Select client + product

    Albert loads the client profile: tone, language, products, region, seasonal context.

  3. Choose a mode

    Suggestions (Albert researches and proposes 3 topics) · Custom topic (editor provides brief) · Comparison (two-product article).

  4. Wait ~8 minutes

    Research → generation → 19-rule validation → auto-correction → WordPress publish. Albert posts the draft link in Slack when ready.

  5. Review and type "approved"

    Editor reviews the WP draft. One word in Slack logs the approval, updates the dashboard, and caches the topic so it's never repeated.

Engineering

What I actually built

The system is a modular Python pipeline — 12 files, each with a single responsibility. The Slack layer has zero business logic. The pipeline has zero Slack imports. This separation means the same agent can be triggered from a scheduler, a CLI, or any future interface without touching the core logic.

The SEO validator — 19 rules in pure Python

Word count, keyword density, H1/H2 presence, meta description length, internal links, image placeholders, FAQ count, em-dashes, and more — all measured in code. The model never self-validates. If the article fails, a surgical correction engine applies fixes in order of blast radius (metadata first, then insertions, then prose rewrites) and re-validates. Max 2 rounds. Remaining failures are surfaced to the editor — never silently swallowed.

The research engine — 4 scrapers running in parallel

Before suggesting topics, Albert runs four scrapers via Playwright concurrently: the client's WordPress blog (to prevent duplicate topics), the client's live navigation (for real internal URLs — never invented), Google's "People Also Ask" (for real search intent), and competitor blogs (for content gap analysis). Total research time is bounded by the slowest single call, not the sum.

The WordPress publisher — Elementor, not raw HTML

Every draft lands in WordPress with a complete Elementor layout — not raw HTML. The editor sees the article exactly as it will look live. Yoast meta description is pre-filled. JSON-LD schemas (BlogPosting, FAQPage, HowTo) are injected. A CTA button links to the client's contact page.

Design Story

Config is the hallucination prevention system

Each of the 27 clients has a JSON profile: name, industry, language, region, tone, vocabulary to use and avoid, products, seasonal calendar, WordPress credentials. Albert can only write facts that exist in this file. No product, URL, or claim is ever invented. The JSON profile isn't just configuration — it's the structural safety layer. By defining a strict boundary of what Albert can say about each client, factual accuracy became a guarantee, not a prompt engineering hope.

Reflection

Four things this project taught me

Separate the model from the logic

The biggest architectural decision was making the LLM call provider-agnostic from day one. The model is called in exactly one function. Everything else — validation, correction, publishing — is deterministic Python. This made the system reliable and testable independent of model behavior. Swap the model, the rest doesn't blink.

Never trust the model to validate itself

All 19 SEO rules are measured in code, not asked of the model. This made every failure deterministic, reproducible, and fixable without touching the prompt. Models are creative writers, not auditors — let each do what they're good at.

The interface is a product decision

Choosing Slack as the interface meant zero onboarding for the content team. They didn't need to learn a new tool. Keeping humans in their existing environment was as important as the technical architecture. Sometimes the best UI is one the user already opens every morning.

Config is the safety layer, not the convenience layer

The JSON profile per client isn't just configuration — it's the hallucination prevention system. Defining a strict boundary of what Albert can say about each client turned factual accuracy from a prompt engineering hope into a structural guarantee.
